# Painter Core - Test Suite

This directory contains the JUnit tests that exercise the Painter native engine (C) and the generated Java bindings.

Key points
- Tests live under `core/src/test/java/net/minestom/painter/`.
- The test suite uses JUnit 5. We also use a small custom test harness annotation `@PaintTest(...)` for tests that need a compiled Painter program executed across generated sections. These tests receive a `ProgramContext` parameter.
- Parser-only tests often use the native parser directly and must free the returned program memory via `PainterParser.freeProgram`.

Running tests (PowerShell)

Run all tests for the `core` module from the repository root:

```powershell
./gradlew :core:test --console=plain
```

Run with more logging:

```powershell
./gradlew :core:test --info
```

Run a single test class or method (use the `--tests` filter):

```powershell
# class
./gradlew :core:test --tests "net.minestom.painter.ParserTest"

# single method in that class
./gradlew :core:test --tests "net.minestom.painter.ParserTest.parses*"
```

About the native library
- The project builds a native shared library (painter.dll on Windows) and places it in `core/build/native/` and also copies it into the test resources so tests can find it at runtime (look under `core/build/resources/test` or `core/build/resources/main` depending on configuration).
- If tests fail with an UnsatisfiedLinkError, verify you have `clang` available (the Gradle build invokes the native toolchain) and rebuild with `./gradlew :core:compileJava` which triggers the native build and jextract generation.

Generated sources
- Java sources generated by `jextract` are under `core/build/generated/sources/jextract/java/` and are automatically added to the `main` source set during the build.

Writing tests

- Parser-only tests should call `PainterParser.parseString` and always free the returned program segment in a finally block. Example helper used across existing tests:

```java
MemorySegment program = null;
try {
    program = PainterParser.parseString(source);
    assertNotNull(program);
} finally {
    if (program != null) PainterParser.freeProgram(program);
}
```

- Execution tests use `@PaintTest(""" ... """)` in the test source; the harness compiles the provided DSL program and injects a `ProgramContext` so the test can call `ctx.generateSection(x,y,z)` to produce `PainterParser.SectionData` for assertions.

- We frequently use parameterized tests for parser-only cases. Example pattern used in this repo:

```java
@ParameterizedTest(name = "{index} {0}")
@MethodSource("validPrograms")
void parses(String program) { assertParses(program); }

static Stream<Arguments> validPrograms() { ... }
```

Troubleshooting
- If tests fail in CI but pass locally, check the native toolchain (clang, jextract) versions.
- If parser tests throw unexpected parse errors, add a small reproducer string to narrow the issue and run only that test with `--tests`.
- If you add new native code, remember to run `./gradlew :core:compileJava` to rebuild the shared lib and regenerate bindings.

Further notes
- Keep tests deterministic: tests that rely on pseudo-random noise use fixed seeds in their `@PaintTest` snippets so results are reproducible.
- When adding programming-model changes (new AST nodes, macros, or tokenizer tokens), update both the C sources in `engine/` and any Java-side usages in tests.
